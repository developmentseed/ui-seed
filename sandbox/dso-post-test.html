Yaml variables

The post configuration goes into the YAML Front Matter. Some of the most important variables:

Base variables

layout: blog
                  category: blog
                  published: true

Date

date: "2015-06-07 16:30"

The date for a post is got from the file name. Only use this variable if the date differs from the one in the file name.
This should be avoided, if possible the file should be renamed.

Author

author: Author name

The post author is defined by using his/her name as it appears under Team.

Title

title: Title of the post

Introduction

introduction: Small introductory sentence

The introduction appears below the title in the post header and in the post card. Should be a small sentence that complements the title.
It’s not a mandatory variable.

Description

description: Description of the post

The description is just used for the <meta> tags for SEO and sharing purposes. If not used will assume the value of introduction.
When there’s no introduction, part of the content will be used.

Topics

topics:
                    - Topic A
                    - Topic B
                    - ...

Posts can have multiple topics. These are used to categorize posts and assist browsing and exploration. Topics appear on list and single pages linking to the respective topic content.

Media

media:
                    card:
                      url: /assets/graphics/content/mapathon-card.jpg
                    cover:
                      url: /assets/graphics/content/mapathon-cover.jpg
                      attribution:
                        author: Marco Antonio
                        url: https://www.flickr.com/photos/38295840@N03/15574324266

The media tag defines the images to use for the post header (cover) and for the post card.

The media.card is mandatory and does not support attribution.

The media.cover is optional and should include the attribution (when pertinent).
This image is mostly decorative and should not be used as a replacement of other content images.
Also, an image used as the post cover should not be used in the post content.

Check the Cover and card images section for instruction on how to prepare the images for preprocessing.

Theme

theme: positive or negative

It’s highly advised to use a negative theme when a cover image is used. By default the positive theme is used.

A complete list of the variables for each content type can be found in the CONTENT_GUIDELINES.md file.

Cover and card images

The raw images must be placed in the src/assets/graphics/content directory in the corresponding folder.
They must have the correct aspect ratio and a size >= the cut size.

Avoid using gifs, as they can’t be processed and increase the visual noise on the list.

  • cover image: To be placed inside cover it will be cut to 1920x960 with a ratio of 2:1
  • card image: To be placed inside card it will be cut to 672x336 with a 2:1 ratio.
  • If you need the same image for card and cover, place it inside cover--card. It will be cut to both sizes.

The url to use can be easily predicted:
- cover image: /assets/graphics/content/[img_name]-cover.[ext]
- card image: /assets/graphics/content/[img_name]-card.[ext]

Headings

The content of a post should not have level 1 headings as these are reserved for the post main title.

Dropcap

The dropcap is the capitalized first letter of a paragraph. It’s not for every occasion, but when the post/page starts with a paragraph that spans at least two full lines it should be added as it has a big impact on the look and feel of the post.

Unlike this example there should be only one dropcap and only at the beginning of the post.

To add it, just place {: .dropcap } after the paragraph:

Lorem ipsum dolor sit amet, ut labore et dolore magna aliqua.
                  Ut enim ad minim veniam, quis nostrud exercitation ullamco
                  laboris nisi ut aliquip ex ea commodo consequat.
                  {: .dropcap }

The dropcap is mandatory be default and its existence is checked by a test that will fail if it is not found.
However, since using it is an editorial choice it can be disabled by adding the # no-dropcap comment to the front matter:

---
                  # no-dropcap
                  layout: blog

This tag just adds the .dropcap class to the element immediately before it, in this case the paragraph.
The same technique can be used to add other classes as can be seen in the lists section.

Content media

Images for content should be placed inside /assets/graphics/content/prose/.
These images are not processed because there isn’t a one size fit them all, so some extra care is needed, specially when it comes to file size:
- Jpg is best suited for photographs and png for graphics.
- Cut to 1500 width (except for full bleed images)
- Compress jpg (70 should be enough). As an alternative you can use compressor.io

Tests will fail if an image is over 500KB If it really needs to be over this size its name can be prefixed with big-- and it will be skipped. Only use when everything else fails.

To ensure correct styling, the content media (image, video, chart…) should be wrapped in a <figure> element with the respective <figcaption> if it exists.

<figure class="align-center">
                    <img src="http://loremflickr.com/1440/360/lego" alt="Image" width="1440" height="360" />
                    <figcaption>Wow, this is a really cool caption!</figcaption>
                  </figure>

It is possible to align media elements by using align-left, align-center, and align-right. Except for centered media, note that text will wrap around it.

It is also possible to use a media element that spans the full width of the page by using the .bleed-full class.

<figure class="bleed-full">
                    <img src="http://loremflickr.com/1440/360/lego" alt="Image" width="1440" height="360" />
                    <figcaption>Wow, this is another cool caption!</figcaption>
                  </figure>

Code blocks

If the code snippet needs to be highlighted use the liquid tag instead of the triple back ticks:

{% highlight [language] %}
                    if (this) that()
                  {% endhighlight %}

Callouts

This is a callout. And it supports markdown and all.

<div class="callout" markdown="1">
                    This is a callout. And it supports **markdown** and all. 
                  </div>

Note that by using markdown="1" on an HTML tag you ensure that any markdown content inside it will be parsed.

Lists

Besides the regular markdown lists there are a couple of extra ones that can be styled with a class.

Check lists

  • Check list item 1
  • Check list item 2
  • Check list item 3
* Check list item 1
                  * Check list item 2
                  * Check list item 3
                  {: .check-list }

Call-to-Action lists

  • Call-to-Action list item 1
  • Call-to-Action list item 2
  • Call-to-Action list item 3
* Call-to-Action list item 1
                  * Call-to-Action list item 2
                  * Call-to-Action list item 3
                  {: .cta-list }

Footnotes lists

  1. Footnote list item 1
  2. Footnote list item 2
  3. Footnote list item 3
1. Footnote list item 1
                  2. Footnote list item 2
                  3. Footnote list item 3
                  {: .footnotes-list }

Definition lists

A regular definition list looks like this:

Term A
Definiton/description A
Term B
Definiton/description B
Term A
                  : Definiton/description A

                  Term B
                  : Definiton/description B

They can also be presented inline with the use of a class:

Twitter
@developmentseed
Email
info@developmentseed.org
Twitter
                  : [@developmentseed](https://twitter.com/developmentseed)

                  Email
                  : [info@developmentseed.org](mailto:info@developmentseed.org)
                  {: .details-list }

Tables

To ensure that tables work on small screens always add a wrapper around them:

instance type process type old-landsat-util new-landsat-util speedup
R3.large non-pansharpened 252.7s 122.05s 2x
R3.large pansharpened 846.9s 349.17s 2.4x
C4.2xlarge non-pansharpened 216.6s 106.67s 2x
C4.2xlarge pansharpened 438s 290s 1.5x 
<div class="table-responsive" markdown="1">

                    instance type | process type| old-landsat-util | new-landsat-util | speedup
                    -----|---- | ----------- | ----------- | -------
                    R3.large | non-pansharpened | 252.7s | 122.05s | **2x**
                    R3.large | pansharpened | 846.9s | 349.17s | **2.4x**
                    C4.2xlarge | non-pansharpened | 216.6s | 106.67s | **2x**
                    C4.2xlarge | pansharpened | 438s | 290s | **1.5x**

                  </div>

Note that the line break between the wrapper div and the markdown is of the utmost importance. Without it the table won’t render correctly.
